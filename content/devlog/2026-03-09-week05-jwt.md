## What the task was

Add authentication to the API. Protected endpoints should reject requests without a valid token.

## What I built

`SecurityController` handles register and login. `TokenUtils` generates and validates JWT tokens. A before-filter in `ApplicationConfig` intercepts every request to protected routes.

Login flow:
1. Client sends `{ username, password }` to `POST /api/auth/login`
2. Server fetches the user, calls `verifyPassword()` with BCrypt
3. If valid, returns a signed JWT token
4. Client sends the token in every subsequent request: `Authorization: Bearer <token>`
5. Before-filter validates the token before the request reaches any controller
```java
public static String generateToken(String username, List<String> roles) {
    return Jwts.builder()
        .setSubject(username)
        .claim("roles", roles)
        .setIssuedAt(new Date())
        .setExpiration(new Date(System.currentTimeMillis() + 86_400_000))
        .signWith(getSigningKey())
        .compact();
}
```

A JWT has three parts separated by dots: `header.payload.signature`. The header says which algorithm was used. The payload carries claims (username, roles, expiry). The signature is a cryptographic hash of the other two parts using a secret key — tampering with the payload breaks the signature.

## JWT vs session-based auth

The previous semester project (Fog Carport) used session-based authentication — the server stores a session object keyed by `currentAdmin`. That works fine for a single server, but doesn't scale horizontally. If you have multiple server instances, they don't share session state.

JWT is stateless — the server stores nothing. The token carries all the information needed to authenticate the request. Any server with the secret key can validate it.

For a REST API, JWT is the right choice.

## Secret key handling

The signing key comes from an environment variable, never hardcoded:
```java
private static final String SECRET_KEY = System.getenv("JWT_SECRET");
```

A hardcoded secret in source code means anyone with access to the repo can forge tokens.
